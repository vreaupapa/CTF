Protocol Analysis

Challenge 1 & 2: Proxy Basics
Objective: Intercept and relay messages between two parties.
Concept: Clear-text Communication & Relay Attacks.

In these introductory levels, the protocol communicated in clear text using a specific format: t: for text, n: for name, and d: for data.

Vulnerability: The protocol lacked any form of authentication or encryption.

Solution: By requesting a message from Alice, we received a standard greeting. Sending this exact payload to Bob allowed us to impersonate the flow and receive the flag.

Takeaway: Clear-text protocols are vulnerable to eavesdropping and simple replay attacks.

The 2 flags are: DawgCTF{PR0T0C0LS_R_3ZPZ}, DawgCTF{CH4NG3_0F_PL4N5}



Challenge 3: Missing Alice
Objective: Manually craft a protocol-compliant message.
Concept: Manual Packet Crafting.

Alice was removed from the scenario, forcing us to communicate with Bob directly.

Vulnerability: Bob only validated the content of the message against a hardcoded script, not the source of the message.

Solution: Following the manual’s script, we manually constructed the payload:
"t:Hello|n:bob|t: this is|n:alice|t: give me the flag"

Takeaway: Protocol strictness (like exact spacing and case sensitivity) is crucial when manually interacting with a backend API via curl

flag: DawgCTF{N0_0N3_3LS3_H0M3}



Challenge 4: Real Security!
Objective: Intercept symmetric keys to decrypt traffic.
Concept: Improper Key Exchange (Symmetric Encryption).

Vulnerability: While the data was encrypted, the secret key (k) and the nonce (d) were transmitted in the clear by Alice.

Solution: 1.  Intercepted Alice's message to get the key (k) and the first nonce (d1).
2.  Intercepted Bob's response to get the encrypted flag (d2).
3.  Used the /util/sym_decrypt utility to combine these values and reveal the flag.

Takeaway: Symmetric encryption is only as secure as the key exchange mechanism. Sending keys over the same channel as the encrypted data defeats the purpose of encryption.

flag: DawgCTF{N0T_S0_S3CR3T_K3Y}


Challenge 5: Is This Real?
Objective: Perform a Man-in-the-Middle (MitM) attack on asymmetric encryption.
Concept: Public Key Replacement Attack.

Vulnerability: The protocol used public keys but lacked a way to verify if a public key actually belonged to the sender (no Certificate Authority/Web of Trust).

Solution: 1.  Generated a custom key pair (pub_X, priv_X).
2.  Intercepted Alice's public key but sent our own pub_X to Bob instead.
3.  Bob, thinking he was talking to Alice, encrypted the flag using our public key.
4.  We decrypted Bob's response using our corresponding priv_X.

Takeaway: Asymmetric encryption requires a Public Key Infrastructure (PKI) to prevent attackers from substituting keys.

flag: DawgCTF{C3RT1F13D_1NS3CUR3}
	



Challenge 9: Oracle
Objective: Exploit a Re-encryption Oracle to bypass nested encryption.
Concept: Decryption Oracle Attack / Padding Oracle Logic.

Vulnerability: Alice acted as an "Oracle." If sent a message encrypted with her key, she would decrypt it and, if it contained a specific destination name (X), she would re-encrypt the inner content for that person.

The "Double Oracle" Strategy:

1.Preparation: Generated a personal certificate (cert_X) to establish a valid identity.
2.Initial Interception: Captured Bob's payload intended for Alice.
3.The Oracle Request: Since Alice required the destination name to be inside the encrypted packet, we used the /asym_encrypt utility to wrap Bob's data and our name (ionut) into a new package encrypted for Alice.
4.The Relay: Sent this "Oracle Package" to Alice. She decrypted her layer, saw it was destined for "ionut," and sent it back to us encrypted with our own key.
5.Final Decryption: Because the flag was nested, we repeated the process or used our private key to strip the final layer, validating the original content from Bob.

Takeaway: A participant that decrypts and re-sends data (even if encrypted for a new recipient) can be used as a tool to bypass security layers they were meant to protect

flag: DawgCTF{ST4R3_1NTO_TH3_VO1D}
